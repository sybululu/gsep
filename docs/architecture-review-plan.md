# GESP 题库系统架构审查与改造方案（Cloudflare 部署版）

## 1. 目标

- 确保所有核心功能可用：答题、题库管理、图片管理、同步到 Firestore。
- 确保 Cloudflare Pages 构建稳定：`npm run build` 必须通过。
- 将“失败后排查”改为“提交前校验 + 运行时兜底”。

## 2. 现状问题分层

### 2.1 展示层（UI）

- 题库切换时，异步加载期间可能出现旧数据闪烁。
- 图片管理面板对图片来源（public vs Firestore）处理不一致。

### 2.2 领域层（题库/题目状态）

- 导入 JSON 后，题目列表和图片 board 状态可能不一致。
- 删除题目时，原题关联图片可能残留在 board 中，造成脏状态。

### 2.3 基础设施层（Firebase/Cloudflare）

- Firestore 图片 ID 若非法会导致写入失败。
- Cloudflare 线上环境中，错误提示不够聚焦时，难以快速定位问题。

## 3. 目标架构

### 3.1 数据流

1. UI 触发（导入、删除、同步）
2. 本地状态归一化（questions 与 board 对齐）
3. 同步前预检（版本/题目/图片）
4. Firestore 写入（图片 -> 题库）
5. 结果反馈（成功/可执行错误）

### 3.2 关键设计原则

- **单一真相**：questions 与 board 必须双向一致。
- **失败前置**：预检不通过，不触发远程写入。
- **降级可用**：图片读取失败时回退静态路径，保证 UI 不崩。
- **部署安全**：避免 Node-only 运行时代码进入前端路径。

## 4. 分步改造清单（已执行）

1. 图片 URL 解析支持 Firestore 文档内容读取，并增加缓存与 fallback。
2. 切换题库时显式进入 loading，避免旧数据误显示。
3. 新增题目时自动补齐 board slot，保证状态一致。
4. 删除题目时回收其图片到未分配区，并清理已删除题的 board key。
5. JSON 导入后自动对齐 board slot，避免后续同步时引用错位。

## 5. 验收标准

- `npm run lint` 与 `npm run build` 通过。
- 切换题库后图片可见（包含 Firestore 存储图）。
- 导入/删除题目后，立即可同步且不出现错位或丢图。
- 预检能在提交前给出可执行错误信息。
